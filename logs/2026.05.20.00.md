# 修复 #43037 Gemma4 tool parser multiple tool calls in single delta

---

## 问题

我们知道对于vllm的前端来说，生成的token相当于是通过一个buffer不断读取，可是buffer的长度并没有限制

这意味着，一个buffer可能包含一个/多个function call

可是，看起来很多parser是用的状态机的模式来匹配，这样的方法只能匹配buffer里面的一个function call

## 修复

通过正则表达式匹配多个function call+状态机

## 感悟

1. 这样的问题应该挺多的，很多parser的实现都是当个function call

2. 这样的修复在ai的帮助下，简单，安全了很多

3. 很难说是用一个很通用的架构来达到这样的目标，还是继续这样每个模型手写parser好

4. 但是遇到这样的问题比较少，我理解遇到这个问题的前提是：
    1. 网络通信请求的间隔时间比较长
    2. GPU的速度比较快
