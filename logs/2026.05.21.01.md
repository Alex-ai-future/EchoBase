# 修复 DeepSeek-V4 Auto-Functionalized 问题学习

## 问题描述

在使用 `torch.compile` 运行 DeepSeek-V4 模型时，出现 `AssertionError: auto_functionalized was not removed` 错误。

## 根因分析

1. **自定义算子**: `fused_deepseek_v4_qnorm_rope_kv_rope_quant_insert` 是 DeepSeek-V4 MLA attention 使用的自定义算子
2. **In-place 修改**: 该算子对 `q` 和 `k_cache` 张量进行了 in-place 修改
3. **torch.compile 行为**: 在 `torch.compile` 过程中，AOTAutograd 将这个算子包装为 `auto_functionalized` 节点
4. **功能化传递未处理**: `FixFunctionalizationPass` 没有处理这个算子，导致节点残留在计算图中，触发 PyTorch 的 assertion 失败

## 修复方案

在 `FixFunctionalizationPass` 中添加对该算子的处理：

```python
elif at_target == torch.ops._C.fused_deepseek_v4_qnorm_rope_kv_rope_quant_insert.default:
    mutated_args = {1: "q", 3: "k_cache"}
    self.defunctionalize(graph, node, mutated_args)
```

## 为什么会包装`auto_functionalized`?

1. 纯函数Ops，不会包装，可以进行DCE，Fusion
2. 但是如果产生了inplace的操作，就需要创建copy

## 为什么需要这个pass

### 标准流程

1. FX Graph会用一个函数来包装，并且产生copy处理的节点
2. Inductor会清除处理副本的节点，还原
3. Dispatch到自己的kernel

### 这里

pytorch不能很好的去掉这个复制的操作，所以vllm通过手动处理，在1，2步骤之间处理

## piecewise

1. 编译器生成fx graph
2. split fx graph 成为sub graph
3. cuda capture / eager

缺点：
1. 切分逻辑很困难，本来就应该可以随意中断和恢复，比如中间加入了个无法trace的函数，切分就很复杂

## breakgraph 

1. 当执行到被 @eager_break_during_capture 装饰的函数就自动eager
2. CPU会接管程序，然后调用这个无法capture的kernel

## 启发

1. 原来的piecewise切分逻辑确实不合理，新的方案更合理
