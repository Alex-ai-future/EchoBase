## 什么是n_group？和TP的关系

1. 传统模型的痛点：庞大的 ‭$W_o$‬ 矩阵
在标准的 Multi-Head Attention（MHA）中，每个 Head 的输出维度是 ‭$d_k$‬（例如 128），如果有 ‭$h=128$‬‭‬ 个 Head，把它们拼起来后的总维度就是 ‭$h \times d_k = 16384$‬‭‬‭‬。
紧接着，模型需要通过一个巨大的输出映射矩阵 ‭$W_o$‬ 将这个万维向量投递回模型的 Hidden Size（假设也是 16384）。
•	传统 ‭$W_o$‬ 参数量： ‭$16384 \times 16384 \approx 2.68$‬‭‬‭‬ 亿参数（这仅仅是单层 Attention 的输出矩阵！）。
•	带来的问题： 在 Decoding 阶段，每个 Token 都要和这个庞大的矩阵做一次 GEMM（矩阵乘法），计算密集（Compute-bound）且消耗大量的带宽。
2. V4 的解法：Grouped Low-Rank 分解
DeepSeek-V4 认为，128 个注意力头捕捉到的信息虽然丰富，但直接用一个铺满的稠密大矩阵去投影，冗余度极高。于是它引入了 Grouped Output Projection：
	1.	切分（Grouping）： 将 128 个 Head 均匀分成 16 个组（Group），每个组包含 8 个 Head。
	2.	第一步投影（Down-Projection）： 每个组单独通过一个低秩矩阵（LoRA 思想），将这 8 个 Head 的输出（‭$8 \times 128 = 1024$‬‭‬‭‬ 维）投影到一个组内隐空间（o_lora_rank = 1024，甚至更低）。
	3.	拼接与最终投影（Up-Projection）： 把这 16 个组压缩后的向量拼接在一起，再通过一个整体的矩阵映射回 Hidden Size。
	DeepSeek-V4-Flash (284B，激活 13B)： 它的 o_group 实际配置为 8 个组（即 o_group = 8）。
•	DeepSeek-V4-Pro (1.6T，激活 49B)： 因为总 Head 数更多、规模更大，它的配置为 16 个组（即 o_group = 16）。

这个组的大小是训练的时候订好的

TP就会把对应的这个投影矩阵给拆分


## 这个bug的原因

如果tp的数量过大，就会导致拆分的一个gpu上一组都没有

## deepgemm_post_process_fp8_weight_block的作用

# 假设 TP=2, wo_a 的形状是 [1024, 512] (每张卡4组)

# 1. 权重是 2D 的
wq.shape = [1024, 512]  # [输入维度, 输出维度]
ws.shape = [8, 4]       # [1024/128, 512/128] 每128x128块一个scale

# 2. 转成 3D 用于 grouped BMM
g = bmm_batch_size = n_local_groups = 4  # 每张卡4组
r = 512 // 4 = 128  # 每组128列
d = 1024  # 输入维度

wq: [1024, 512] → [g, r, d] = [4, 128, 1024]  # 转置后
ws: [8, 4] → [g, r/128, d/128] = [4, 1, 8]

# 3. 转换 scale 格式给 DeepGEMM
dg_ws = transform_sf_into_required_layout(ws, ...)
# 把 scale 排成 DeepGEMM 要求的内存对齐格式

# 4. 返回
return wq, dg_ws  # 用于后续的 fp8_einsum 计算
